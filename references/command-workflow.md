# 命令工作流

## `/sg-task create <任务名称>`

用途：创建任务主档，并绑定涉及仓库与当前分支。

规则：

1. 如果配置文件不存在，先初始化全局仓库配置
2. 初始化时为每个仓库设置 `name / key / type / path`
3. 读取配置文件，展示可选仓库
4. 用户选择涉及仓库
5. 对每个选中仓库读取当前分支
6. 某仓库没有可绑定分支时，要求先切换或创建分支
7. 生成任务目录和 `meta.md`
8. 按需建议创建 `product.md`、`development.md`、`api.md`

输出重点：

- 任务已创建
- 绑定了哪些仓库与分支
- 接下来建议补哪些文档

## `/sg-task show`

用途：按“当前仓库 + 当前分支”反查任务，只显示摘要。

规则：

1. 确定当前仓库路径和当前分支
2. 遍历 `.tasks/*/meta.md`
3. 用 `repositories[].path + repositories[].branch` 匹配任务
4. 命中 1 个任务则展示摘要
5. 命中 0 个任务则提示当前分支未绑定任务
6. 命中多个任务则提示数据异常

## `/sg-task show --load`

用途：恢复任务工作态。

规则：

1. 先执行 `/sg-task show` 定位任务
2. 读取 `meta.md`
3. 根据 `context_refs` 精确读取关键区块
4. 只有用户明确要求时，才继续深读完整原文

## `/sg-task doc product`

重点补这些区块：

- 任务背景
- 目标与成功指标
- 功能需求清单
- 改动范围
- 不在本次范围事项
- 验收标准

## `/sg-task doc development`

重点补这些区块：

- 当前状态
- 进行中
- 待开始
- 已完成
- 测试记录
- 阻塞问题
- 下一步计划

## `/sg-task doc api`

只在存在明确联调边界时创建或补全。

## `/sg-task add-repo`

用途：给当前任务补挂一个已存在于全局配置中的仓库。

规则：

1. 先定位当前任务
2. 从配置中选择尚未加入任务的仓库
3. 读取该仓库当前分支
4. 分支存在时写入 `meta.md`
5. 分支不可绑定时拒绝写入

## `/sg-task remove-repo`

用途：从当前任务移除仓库映射。

## `/sg-task progress`

用途：读取 `development.md`，展示进度摘要。

优先读取：

- 当前状态
- 进行中
- 待开始
- 已完成
- 阻塞问题
- 下一步计划

不要把测试模板细项直接混入任务完成度。

## `/sg-task complete`

用途：轻量收尾。

规则：

1. 定位当前任务
2. 将 `meta.md` 状态改为 `completed`
3. 提示补最终测试结果或交付说明

## `/sg-task add-repo-config`

用途：给整个项目新增全局仓库配置。

规则：

1. 新增仓库时设置 `name / key / type / path`
2. `name` 在这里确定，后续任务创建直接复用
3. 不要把仓库命名流程放到 `create` 的日常路径里重复执行

## `/sg-task sync`

用途：当 `auto_commit: false` 时，手动将 `.tasks` 目录中的文档变更提交到 Git。
